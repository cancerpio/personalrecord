## Context

使用者在首頁（Dashboard）需要掌握其「每週訓練容積」（Weekly Training Volume，定義為該週所有動作的組數 * 次數 * 重量之加總）以及其「訓練趨勢」（本週容積與歷史每週平均值相比的變化：上升、持平或下降）。所有訓練資料已由前端 `sessionStore.js` 透過 API `getSessions()` 載入，並存在 `state.sessions` 中。因此，計算可以直接在前端進行，不需要額外修改後端資料庫架構或 API。

## Goals / Non-Goals

**Goals:**
- 在 `sessionStore.js` 中新增計算「每週訓練容積」、「歷史每週平均值」與「容積趨勢」的 getters/actions。
- 計算以週一為一週起點（ISO 8601 週），對所有訓練動作進行統計。
- 在首頁（`DashboardView.vue`）以高質感的 UI 展示當週總訓練容積與趨勢。
- 趨勢規則：當週容積與歷史每週平均值相比，高於 5% 為上升，低於 5% 為下降，正負 5% 以內為持平。

**Non-Goals:**
- 排除修改後端資料庫的資料表欄位或資料模型，皆以現有的 `sets`、`reps`、`weight` 進行前端動態彙整。
- 排除以圖表（Highcharts）繪製容積歷史趨勢，僅以文字、圖標及顏色（上升為綠色，下降為紅色，持平為橙色）進行顯著提示。

## Decisions

### 1. 訓練週的劃分演算法 (Weekly Grouping)
- **選擇方案**：利用 UTC 時間，將 `YYYY-MM-DD` 格式的日期換算為對應週的「星期一日期字串」（`YYYY-MM-DD`）。
- **理由**：避免時區位移問題，確保所有同週的訓練日誌不論在何時區讀取，均會被正確歸類至該週的星期一。

### 2. 歷史平均容積的排除規則
- **選擇方案**：計算歷史每週平均訓練容積時，應「排除當前尚未結束的這一週」。
- **理由**：當週若只過了一天或正在進行中，容積通常會遠低於已完成整週的歷史平均值，如果納入計算會拉低平均，且直接比對會造成無意義的「下降」趨勢。

### 3. UI 呈現位置與風格
- **選擇方案**：在 `DashboardView.vue` 的標題下方、Sparkline 行上方，以一個寬度佔滿、設計具備毛玻璃效果的卡片（類似 `CycleStatus.vue` 的風格）呈現「每週訓練總容積」。
- **理由**：首頁上方是使用者開啟 App 時視線的第一落點，能讓使用者最快得知自己當週的運動進度。

## Risks / Trade-offs

- **[Risk]** 當使用者歷史紀錄極少（如只有本週資料）時，歷史平均為 0。
  - *Mitigation*: 當歷史無其他週資料時，趨勢顯示為「無歷史數據較對」或「首週訓練中」（趨勢狀態定為 `none`），不顯示誤導的上升或下降箭頭。
